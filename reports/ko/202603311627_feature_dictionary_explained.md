# Feature Dictionary 설명

## Purpose

이 보고서는 Advanced XGBoost 기능 열의 의미를 일반 언어로 설명합니다.

목표는 machine learning에 깊이가 없는 독자가 모델 입력을 이해할 수 있도록 만드는 것입니다. 여기 설명은 모델이 나중에 로컬 클라우드 환경에 맞게 tuning되는 최종 프로젝트 마일스톤에 특히 유용합니다.

## 이 기능을 읽는 방법

기능은 주어진 시간 창에서 작업의 측정되거나 파생된 속성 중 하나입니다.

모델은 전체 작업 "스토리"를 직접적으로 볼 수 없습니다. 다음과 같은 많은 열이 표시됩니다.

- 얼마나 많은 CPU가 사용되었는지
- 얼마나 많은 메모리가 사용되었는지
- 작업이 얼마나 오래되었는지
- 주변에서 얼마나 많은 사건이 일어났는지
- 기계가 얼마나 혼잡한지
- 최근 근처에서 유사한 고장이 발생했는지 여부

그런 다음 해당 신호를 결합하여 다음 `5m`, `15m`, `30m`, `45m` 또는 `60m`의 실패 위험을 추정합니다.

## 중요 참고 사항

- `priority` 및 `scheduling_class`은 모델 자체 출력이 아니라 워크로드 스케줄러의 메타데이터입니다.
- `avg_mem_to_request_ratio` 등의 비율은 "요청한 것 대비 얼마나 사용되었는지"를 나타냅니다.
- `_lag_1`, `_delta_1` 또는 `_roll3_mean`로 끝나는 기능은 동일한 작업의 최근 과거에서 파생된 시계열 기능입니다.
- `_is_missing`으로 끝나는 기능은 소스 값이 누락되었는지 여부를 표시하는 이진 표시기입니다.

## 기능 그룹

### 1. 기본 사용 기능

이는 현재 관찰 창 내에서의 직접적인 리소스 사용량을 설명합니다.

| 기능 | 의미 |
| --- | --- |
| `usage_window` | 행의 사용량 관찰 창 길이 또는 크기 |
| `avg_cpu` | 해당 기간 동안 사용된 평균 CPU |
| `max_cpu` | 해당 기간 동안 사용된 최대 CPU |
| `avg_mem` | 해당 기간 동안 사용된 평균 메모리 |
| `max_mem` | 해당 기간 동안 사용된 최대 메모리 |
| `avg_cpu_utilization` | 사용률로 정규화된 평균 CPU 사용량 |
| `max_cpu_utilization` | 창의 최고 CPU 사용률 |
| `avg_mem_utilization` | 창의 평균 메모리 사용률 |
| `max_mem_utilization` | 창의 최대 메모리 사용률 |

운영상의 의미:

- 작업이 가볍게 로드되는지, 지속적으로 로드되는지 또는 최고점/급등을 표시하는지 알려줍니다.

### 2. 요청된 리소스 및 스케줄러 메타데이터

이는 작업이 요청한 내용과 스케줄러가 작업을 분류한 방법을 설명합니다.

| 기능 | 의미 |
| --- | --- |
| `req_cpu` | 작업에서 요청한 CPU |
| `req_mem` | 작업에서 요청한 메모리 |
| `priority` | 작업의 스케줄러 우선순위 수준 |
| `scheduling_class` | 작업에 대한 스케줄러 클래스 또는 정책 버킷 |

운영상의 의미:

- `priority`은 일반적으로 워크로드가 스케줄러에게 얼마나 중요하거나 긴급한지를 반영합니다.
- `scheduling_class`은 일반적으로 스케줄러가 사용하는 배치/처리 정책의 종류를 반영합니다.
- 긴급하거나 특수한 작업은 백그라운드 또는 일괄 작업과 다르게 동작할 수 있으므로 이는 중요할 수 있습니다.

### 3. 수명 주기 및 이벤트 기록 기능

이는 작업이 해당 수명의 어느 위치에 있는지, 얼마나 많은 이벤트 활동이 축적되었는지를 설명합니다.

| 기능 | 의미 |
| --- | --- |
| `event_count` | 작업에 연결된 결합된 이벤트 레코드 수 |
| `task_age_us` | 작업의 첫 번째 관찰된 이벤트부터 현재 기간 종료까지 경과된 시간 |
| `observed_failure_by_window` | 현재 창이 끝나기 전에 이미 오류와 유사한 터미널 패턴이 표시되었는지 여부 |
| `machine_recent_failure_count_12` | 이전 12개의 관찰 창에 걸쳐 동일한 시스템의 최근 실패 횟수 |
| `collection_recent_failure_count_12` | 이전 12개 창에 대한 동일한 컬렉션의 최근 실패 횟수 |
| `machine_recent_terminal_count_12` | 이전 12개 창에 걸쳐 동일한 시스템의 최근 터미널 이벤트 수 |
| `collection_recent_terminal_count_12` | 이전 12개 창에 대한 동일한 컬렉션의 최근 터미널 이벤트 수 |

운영상의 의미:

- `task_age_us` 답변: "지금 이 작업은 얼마나 오래 되었습니까?"
- `event_count` 답변: "이 작업과 관련하여 얼마나 많은 스케줄러/이벤트 활동이 발생했습니까?"
- '최근 고장'과 '최근 단말기' 기능 답변: "이 동네가 최근에 벌써 불안했나요?"

### 4. 기계 용량 특징

이는 기계의 사용 가능한 용량을 설명합니다.

| 기능 | 의미 |
| --- | --- |
| `machine_cpu` | 할당된 시스템의 CPU 용량 |
| `machine_mem` | 할당된 시스템의 메모리 용량 |

운영상의 의미:

- 이를 통해 모델은 작업 크기를 기계 크기와 비교할 수 있습니다.

### 5. 자원적합성 비율 기능

이는 사용량이나 요청을 요청된 용량 또는 사용 가능한 용량과 비교합니다.

| 기능 | 의미 |
| --- | --- |
| `cpu_request_ratio` | 요청된 CPU를 시스템 CPU로 나눈 값 |
| `mem_request_ratio` | 요청된 메모리를 시스템 메모리로 나눈 값 |
| `avg_cpu_to_request_ratio` | 평균 CPU 사용량을 요청된 CPU로 나눈 값 |
| `avg_mem_to_request_ratio` | 요청된 메모리로 나눈 평균 메모리 사용량 |
| `max_cpu_to_request_ratio` | 최고 CPU 사용량을 요청된 CPU로 나눈 값 |
| `max_mem_to_request_ratio` | 최대 메모리 사용량을 요청된 메모리로 나눈 값 |

운영상의 의미:

- `1` 근처 또는 그 이상의 값은 종종 작업이 요청된 수준에 가깝거나 그 이상을 사용하고 있음을 의미합니다.
- 요청이 부족한 리소스나 워크로드 불일치를 감지하는 데 유용합니다.

### 6. 헤드룸 및 스파이크 기능

이는 느슨함과 폭발성을 나타냅니다.

| 기능 | 의미 |
| --- | --- |
| `cpu_headroom` | 머신 CPU에서 평균 작업 CPU 사용량을 뺀 |
| `mem_headroom` | 머신 메모리에서 평균 작업 메모리 사용량을 뺀 |
| `cpu_spike_gap` | 최고 CPU에서 평균 CPU를 뺀 |
| `mem_spike_gap` | 최고 메모리에서 평균 메모리를 뺀 |

운영상의 의미:

- `headroom` 답변: “공간이 얼마나 남았나요?”
- `spike_gap` 답변: "정상 수준에 비해 이 작업이 얼마나 폭발적인가요?"

### 7. 머신 수준 및 컬렉션 수준 크라우딩 기능

이는 동일한 기간 동안 환경이 얼마나 혼잡한지를 설명합니다.

| 기능 | 의미 |
| --- | --- |
| `machine_task_count_window` | 현재 창에서 동일한 시스템의 작업 수 |
| `collection_task_count_window` | 현재 창에서 동일한 컬렉션에 있는 작업 수 |
| `machine_window_avg_cpu_sum` | 동일한 창에 있는 동일한 시스템의 작업 전체에 대한 평균 CPU 합계 |
| `machine_window_avg_mem_sum` | 동일한 창에 있는 동일한 컴퓨터의 작업 전체에 대한 평균 메모리 합계 |
| `machine_window_avg_cpu_load_ratio` | 머신 용량으로 정규화된 머신 수준 평균 CPU 로드 |
| `machine_window_avg_mem_load_ratio` | 머신 용량으로 정규화된 머신 수준의 평균 메모리 로드 |
| `collection_window_avg_cpu_sum` | 동일한 컬렉션 및 창에 있는 작업 전체의 평균 CPU 합계 |
| `collection_window_avg_mem_sum` | 동일한 컬렉션 및 창에 있는 작업 전체의 평균 메모리 합계 |

운영상의 의미:

- 다음과 같은 질문에 답합니다.
  - “호스트가 붐비나요?”
  - "지금 이 컬렉션 전체가 바쁘나요?"
  - "실패 위험은 로컬 작업 동작에서 발생합니까, 아니면 혼잡한 환경에서 발생합니까?"

### 8. 가변성 기능

이는 작업 자체 시계열 내의 최근 불안정성 또는 변동성을 설명합니다.

| 기능 | 의미 |
| --- | --- |
| `avg_cpu_roll6_std` | 지난 6번의 관찰에 대한 평균 CPU의 표준 편차 |
| `avg_mem_roll6_std` | 지난 6번의 관찰에 대한 평균 메모리의 표준 편차 |
| `avg_cpu_utilization_roll6_std` | 지난 6번의 관찰에 대한 평균 CPU 사용률의 표준 편차 |
| `avg_mem_utilization_roll6_std` | 지난 6번의 관찰에 대한 평균 메모리 사용률의 표준 편차 |

운영상의 의미:

- 값이 높으면 작업이 불안정하거나 시간이 지남에 따라 변동한다는 의미입니다.

### 9. 짧은 역사의 시간적 특징

이는 현재 관찰을 작업의 최근 과거와 비교합니다.

#### 지연 기능

| 기능 | 의미 |
| --- | --- |
| `avg_cpu_lag_1` | 이전 창의 평균 CPU |
| `max_cpu_lag_1` | 이전 창의 최대 CPU |
| `avg_mem_lag_1` | 이전 창의 평균 메모리 |
| `max_mem_lag_1` | 이전 창의 최대 메모리 |
| `avg_cpu_utilization_lag_1` | 이전 창의 평균 CPU 사용률 |
| `max_cpu_utilization_lag_1` | 이전 창의 최대 CPU 사용률 |
| `avg_mem_utilization_lag_1` | 이전 창의 평균 메모리 사용률 |
| `max_mem_utilization_lag_1` | 이전 창의 최대 메모리 사용률 |

#### 델타 기능

| `avg_mem_delta_1` | 현재 평균 메모리에서 이전 평균 메모리를 뺀 |

| `max_cpu_utilization_delta_1` | 현재 최대 CPU 사용률에서 이전 최대 CPU 사용률을 뺀 |
| `avg_mem_utilization_delta_1` | 현재 평균 메모리 사용률에서 이전 평균 메모리 사용률을 뺀 |
| `max_mem_utilization_delta_1` | 현재 최대 메모리 사용량에서 이전 최대 메모리 사용량을 뺀 |

| 기능 | 의미 |
| --- | --- |

| `max_cpu_roll3_mean` | 지난 3번의 관찰에 대한 최대 CPU의 평균 |
| `avg_mem_roll3_mean` | 지난 3번의 관찰에 대한 평균 기억력 |
| `max_mem_roll3_mean` | 지난 3번의 관찰에 대한 최대 메모리의 평균 |
| `avg_cpu_utilization_roll3_mean` | 지난 3번의 관찰에 대한 평균 CPU 사용률 |
| `max_cpu_utilization_roll3_mean` | 지난 3번의 관찰에 대한 최대 CPU 사용률의 평균 |
| `avg_mem_utilization_roll3_mean` | 지난 3번의 관찰에 대한 평균 메모리 사용률 |
| `max_mem_utilization_roll3_mean` | 지난 3번의 관찰에 대한 최대 메모리 사용률의 평균 |

운영상의 의미:

- 지연 기능은 지금 직전에 무슨 일이 일어났는지 모델에 알려줍니다.
- 델타 기능은 작업이 상승하는지, 하락하는지, 급격하게 변화하는지 알려줍니다.
- 롤링은 노이즈를 완화하고 단기적인 추세를 보여주는 것을 의미합니다.

### 10. 실종 플래그

다음과 같은 이진 표시기입니다.

- `avg_cpu_is_missing`
- `req_mem_is_missing`
- `cpu_request_ratio_is_missing`
- `collection_recent_failure_count_12_is_missing`

의미:

- `1`은 해당 행에 대한 소스 기능이 누락되었음을 의미합니다.
- `0`는 소스 기능이 존재했음을 의미합니다.

운영상의 의미:

- 때로는 "누락" 자체가 유익할 때도 있습니다.
- 저장된 기본 모델에서 이러한 플래그의 대부분은 중요도가 거의 0에 가까웠는데, 이는 현재 데이터 조각이 상당히 완전했거나 원시 값 열에 이미 사용 가능한 대부분의 정보가 포함되어 있음을 의미합니다.

## 실제로 가장 중요한 기능

저장된 기본 모델에서 운영상 가장 중요한 기능은 대부분 다음과 같습니다.

- `task_age_us`
- `event_count`
- `observed_failure_by_window`
- `scheduling_class`
- `priority`
- `collection_recent_failure_count_12`
- `req_cpu`
- `avg_mem_to_request_ratio`

이는 모델이 주로 다음을 요구한다는 것을 의미합니다.

1. 작업은 몇 살부터 시작됩니까?
2. 주변에서 이미 얼마나 많은 이벤트 활동이 일어났나요?
3. 그 회사나 그 이웃이 이미 실패의 조짐을 보였습니까?
4. 어떤 스케줄러 체제에서 실행되고 있습니까?
5. 리소스 사용량이 요청한 것과 일치하지 않는 것처럼 보입니까?

## 미래의 로컬 클라우드 마일스톤을 위한 매핑 지침

로컬 클라우드 환경에 적응할 때 가장 안전한 기능 우선순위는 다음과 같습니다.

### 보존 우선순위가 가장 높음

- 작업/창 타임스탬프
- CPU 및 메모리 사용량
- CPU 및 메모리 요청 또는 제한
- 우선순위/예약 메타데이터가 있는 경우
- 기계 용량
- 단말 이벤트/장애 내역

### 가능하면 보관하는 것이 좋습니다

- 머신 수준 동시 작업 수
- 수집 또는 서비스 수준 집계 로드
- 작업별 최근 시간 기록

### 정확한 등가물이 존재하지 않는 경우 대략적인 추정이 허용됩니다.

- Borg별 예약 클래스 버킷
- 로컬 클라우드가 서비스/작업/애플리케이션 그룹화를 대신 사용하는 경우 Borg 컬렉션 그룹화
- 로컬 플랫폼이 다른 수명주기 이벤트를 발생시키는 경우 이벤트 카운트 의미론

## 마지막 주의

이러한 기능 이름은 Borg 스타일 데이터 조인으로 구축되었기 때문에 의미가 있습니다. 로컬 클라우드로 이동하는 경우:

- 원래 필드 이름이 반드시 필요하지 않고 작동 의미를 동일하게 유지하십시오.
- 가능한 경우 방향성과 단위를 유지합니다.
- 모든 기능 대체 사항을 주의 깊게 문서화하세요.

예를 들면:

- Borg `collection_id`은 로컬 클라우드 서비스, 배포, 작업 또는 테넌트 식별자에 매핑될 수 있습니다.
- Borg `priority`는 대기열 우선순위, QoS 계층 또는 예약 정책에 매핑될 수 있습니다.
- Borg `event_count`은 재시작 횟수, 수명 주기 전환 횟수 또는 오케스트레이션 이벤트 횟수에 매핑될 수 있습니다.

가장 강력한 전송 전략은 정확한 스키마 표현이 아닌 의미론을 보존하는 것입니다.
